# 📖 学习笔记: 基于 ModelScope 的 Gemma-4 单卡 LoRA 微调实战

## 📝 导语

本笔记记录了在 Linux 环境下单卡微调大语言模型（LLM）的完整实践。此次实验选择了 `google/gemma-4-E4B-it` (Instruction-tuned) 模型，在 `AI-ModelScope/emotion` 数据集上进行情绪分类任务的 LoRA 微调。

实验的核心目标是探索**完全脱离 Hugging Face Hub 的本地化训练链路**，并验证在有限算力下（单卡、兼容 AMD ROCm 环境）的高效微调策略。

## 🛠️ 核心技术栈与环境约束

* **基础大模型**: `google/gemma-4-E4B-it` (自带官方 Chat Template)


* **数据集**: `AI-ModelScope/emotion` (包含 sadness, joy, love, anger, fear, surprise 6 个类别)


* **训练框架**: `PyTorch`, `Transformers`, `PEFT`, `TRL`

* **数据源与模型源**: 魔搭 ModelScope


* **环境适配**: 针对 AMD ROCm 环境，移除了 `bitsandbytes` 4bit 量化，使用纯 BF16/FP16 训练，并采用 `adamw_torch` 优化器规避兼容性问题。



## 💡 核心填坑与架构考量

### 1. 彻底的数据集本地化闭环

在实际开发中，直接使用高层 API 往往会遇到版本错配问题（如 `MsDataset` 与 `datasets` 库的底层兼容报错）。本方案采用了一种更鲁棒的工程实现：
先通过 `dataset_snapshot_download` 将整个仓库拉取到本地，再利用 `datasets.load_dataset("parquet", ...)` 直接从底层文件读取。这不仅解决了网络和授权依赖，也提升了数据流转的透明度。

```python
# 规避版本报错的优雅加载方式
dataset_dir = dataset_snapshot_download(MODELSCOPE_DATASET_ID, cache_dir="./datasets")
raw_dataset = load_dataset(
    "parquet",
    data_files={
        "train": _parquet_files_for("train"),
        "validation": _parquet_files_for("validation"),
        "test": _parquet_files_for("test"),
    },
)

```

### 2. Prompt 工程与格式对齐

为契合 Instruction-tuned 模型的特性，将原本扁平的分类数据转化为对话格式（System-User-Assistant）。系统提示词严格限制了输出格式：

> *"Read the user's text and answer with exactly one label. Only choose from: sadness, joy, love, anger, fear, surprise."*
> 

### 3. LoRA 策略与显存管理

采用 `target_modules="all-linear"` 注入适配器，训练参数占比约 0.63%（约 5000 万参数）。在单卡配置下，结合 `gradient_accumulation_steps=4` 和 `gradient_checkpointing=True` 成功实现了显存与批次大小（等效 Batch Size = 16）的平衡。

## 🚀 总结与后续演进路线

本次单卡 LoRA 实验不仅验证了 Gemma-4 的微调潜力，也跑通了一套纯粹的本地化开发工作流。为了将此模型从实验环境推向实际工程落地，接下来的演进计划包括：

1. **API 接口标准化封装**：基于 vLLM 或 TGI 加载合并后的 LoRA 模型，并构建一层 API 网关，将其严格封装为兼容 `OpenAI /v1/chat/completions` 格式的 RESTful 接口，无缝对接现有的业务系统与开发标准。
2. **RAG 系统深度集成**：这种高精度的情绪分类模型非常适合作为 RAG (Retrieval-Augmented Generation) 架构的**前置意图路由器**。通过分析用户 Query 的情绪状态，系统可以动态调整 RAG 检索库的权重或改变最终生成的 prompt 语气（例如对 anger 情绪提供更具安抚性的事实依据），从而在不增加开发人员日常维护负担的前提下，显著提升 RAG 知识问答的智能化与柔性。
3. **算力拓展**：当前单卡方案逻辑已跑通，后续可借助集群环境引入分布式并行策略，进一步扩展训练规模。
